# Research: External precedents for a Pixi daemon

## Summary
Large developer/build tools generally keep the CLI thin and move expensive graph/config/file-state work into a long-lived daemon. Buck2, Bazel, Pants, and Gradle all favor local socket/RPC protocols and in-memory caches, with lifecycle guards for version/options/workspace changes; none of the mainstream tools use shared memory as the primary control plane. For Pixi, the strongest precedent is a lazily-started per-workspace daemon that watches relevant files, owns cached state, serves multiple CLI commands over a simple local IPC API, and can be killed/restarted safely.

## Findings
1. **Daemon vs CLI split: keep CLI as launcher/terminal proxy; daemon owns expensive state.** Buck2 states that most commands require the daemon and that “most work is done by the daemon process”; the CLI connects to a gRPC service, verifies version, and starts/restarts the daemon if needed. Bazel similarly has a C++ client that locates/starts the server, forwards the command over gRPC, streams output, handles Ctrl-C, and returns the server-provided exit code. Gradle’s client connects to or starts a daemon, sends command-line args/project dir/environment, and streams output; the daemon resolves dependencies, executes scripts/tasks, and caches state. [Buck2 lifecycle](https://github.com/facebook/buck2/blob/main/app/buck2_daemon/daemon_lifecycle.md), [Bazel codebase architecture](https://bazel.build/versions/9.1.0/contribute/codebase), [Gradle daemon](https://docs.gradle.org/current/userguide/gradle_daemon.html)
   - **Pixi implication:** implement `pixi` as a robust client: parse early/help/version locally, discover workspace, locate/start daemon, forward command, stream stdout/stderr/events, propagate exit code, and translate interrupts/cancel. Put manifest parsing, environment solving/checking metadata, lockfile/index state, and FS invalidation in daemon memory.

2. **Lifecycle is lazy-started, long-lived, but aggressively restarted on incompatibility.** Buck2 starts `buckd` on first command for the current project and reuses it; clients verify same Buck version and restart on mismatch. Bazel servers stop after inactivity, default 3 hours, and the client restarts if version/startup options/workspace do not match. Gradle daemons are long-lived, idle-stop after 3 hours or low memory, and compatibility requires exact Gradle version/JVM args/environment. Pants has restart triggers for fingerprinted options, daemon invalidation globs, and max memory. [Buck2 daemon](https://buck2.build/docs/concepts/daemon/), [Bazel client/server](https://github.com/bazelbuild/bazel/blob/master/site/en/run/client-server.md), [Gradle daemon](https://docs.gradle.org/current/userguide/gradle_daemon.html), [Pants global options](https://www.pantsbuild.org/dev/reference/global-options)
   - **Pixi implication:** lazy start is enough; avoid a system service. Track a daemon compatibility fingerprint: Pixi binary/version, workspace root, relevant config files, global settings affecting solve/install semantics, selected platform, and cache schema. Add `pixi daemon status`, `pixi daemon stop/kill`, idle timeout, memory cap, and restart-on-panic/version mismatch.

3. **Per-workspace/per-project daemons are the dominant fit for workspace-specific state.** Buck2 defaults to one daemon per project root, with an isolation directory for multiple daemons in one project. Bazel associates each server instance with a single workspace/output base; multiple workspaces get multiple output bases and server processes. Pants currently has at most one `pantsd` per workspace/buildroot. Gradle is more global/user-home oriented and can reuse compatible daemons across builds, but compatibility/version/JVM constraints can spawn multiple daemons; Watchman is the notable global per-user file-watching service. [Buck2 daemon](https://buck2.build/docs/concepts/daemon/), [Bazel client/server](https://github.com/bazelbuild/bazel/blob/master/site/en/run/client-server.md), [Pants global options](https://www.pantsbuild.org/dev/reference/global-options), [Watchman socket interface](https://facebook.github.io/watchman/docs/socket-interface)
   - **Pixi implication:** choose **one daemon per Pixi workspace root** rather than one global daemon. Pixi’s hot state is workspace-specific (`pixi.toml`, lockfile, environments, features, task graph), so per-workspace bounds memory, simplifies invalidation, and avoids cross-repo correctness hazards. A global helper only makes sense later for shared package/index caches or watching infrastructure, not as the primary workspace state owner.

4. **Filesystem watching is a core daemon use case, but must be paired with explicit invalidation.** Buck2’s daemon monitors the project filesystem and respects ignore settings. Pantsd listens to filesystem events and keeps a warm graph; Pants can disable FS watching only if pantsd/loop are not enabled. Gradle uses file-system watching to determine rebuild work before a build. Watchman provides the strongest dedicated precedent: a single per-user service, Unix-domain socket API, subscriptions, and consistency mechanisms around asynchronous FS notifications. [Buck2 daemon](https://buck2.build/docs/concepts/daemon/), [Pants daemon](https://v1.pantsbuild.org/architecture_pantsd.html), [Gradle daemon](https://docs.gradle.org/current/userguide/gradle_daemon.html), [Watchman socket interface](https://facebook.github.io/watchman/docs/socket-interface)
   - **Pixi implication:** daemon should watch only inputs that affect command latency/correctness: `pixi.toml`, lockfiles, config, task files, relevant environment directories, and possibly project source globs only when commands require them. Honor `.gitignore`/Pixi ignores where appropriate, and include a cheap explicit rescan/fingerprint path on each request to handle missed events.

5. **IPC precedent: local sockets + framed protocol/RPC; not shared memory for commands.** Buck2 exposes a gRPC daemon API; Bazel uses gRPC `CommandServer.run/cancel/ping`; Gradle uses a local socket connection; Watchman uses Unix-domain sockets with JSON or BSER PDUs. Pants’ legacy daemon used a Nailgun/Pailgun socket protocol to spawn invocations and pass environment/args/stdin/stdout. [Buck2 lifecycle](https://github.com/facebook/buck2/blob/main/app/buck2_daemon/daemon_lifecycle.md), [Bazel command_server.proto](https://github.com/bazelbuild/bazel/blob/master/src/main/protobuf/command_server.proto), [Gradle daemon](https://docs.gradle.org/current/userguide/gradle_daemon.html), [Watchman socket interface](https://facebook.github.io/watchman/docs/socket-interface), [Pants daemon](https://v1.pantsbuild.org/architecture_pantsd.html)
   - **Pixi implication:** use Unix-domain sockets on Unix/macOS and named pipes or loopback TCP on Windows, with a small protobuf/JSON/MessagePack schema. Include `status`, `run`, `cancel`, `shutdown`, and maybe `watch-health` APIs. Prefer request/response plus streaming events over ad-hoc stdout parsing.

6. **Shared memory is fast but appears inappropriate for Pixi’s first daemon design.** Rust IPC benchmarks show shared memory can beat sockets by large margins in microbenchmarks, but requires unsafe code, manual synchronization, fixed layouts/sizing, and platform-specific behavior; one Rust comparison explicitly warns it is hard to make production-safe unless latency is absolutely critical. The mainstream build/dev daemons above use sockets/RPC, not shared memory, even when performance-critical. [IPC in Rust comparison](https://3tilley.github.io/posts/simple-ipc-ping-pong/), [rusty-comms benchmark](https://github.com/redhat-performance/rusty-comms), [Bazel client/server](https://github.com/bazelbuild/bazel/blob/master/site/en/run/client-server.md)
   - **Pixi implication:** do **not** use shared memory for daemon control or cached-state access initially. The target win is avoiding ~500ms repeated rebuild/check work, while socket overhead is microseconds to low milliseconds and operationally simpler. Consider shared memory only later for very large binary payload transfer or cross-process content-addressed cache reads after profiling.

7. **Concurrency support varies; single-flight is common and simpler.** Bazel says each server handles at most one invocation at a time; concurrent invocations block or fail-fast. Buck2 supports concurrent invocations with the same version but restarts/kill can disconnect them. Pants notes pantsd does not support parallel runs and has a timeout when multiple invocations contend; enabling Pants concurrency starts concurrent runs without pantsd. [Bazel client/server](https://github.com/bazelbuild/bazel/blob/master/site/en/run/client-server.md), [Buck2 lifecycle](https://github.com/facebook/buck2/blob/main/app/buck2_daemon/daemon_lifecycle.md), [Pants global options](https://www.pantsbuild.org/dev/reference/global-options)
   - **Pixi implication:** start with one mutating command at a time per workspace daemon, while allowing cheap read-only queries/status concurrently if safe. This prevents solve/install/environment mutations racing. Add user-facing behavior: wait with timeout, `--no-daemon`, or fail-fast.

## Gaps
- Public Buck2/Bazel docs do not clearly document every low-level transport detail across platforms, especially Windows named-pipe behavior; source inspection would be needed for exact implementation patterns.
- No direct evidence found that mainstream build daemons evaluated and rejected shared memory for their client protocol; the conclusion is inferential from their documented socket/RPC designs plus Rust IPC tradeoffs.
- Pixi-specific command taxonomy is needed to decide which commands are safe read-only concurrent requests versus serialized mutating requests.
