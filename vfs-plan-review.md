## Review

Context note: `e:/projects/pixi/plan.md` and `e:/projects/pixi/progress.md` were not present in this checkout, so this review is based on the task statement plus the current `pixi_compute_fs` / compute-engine code.

- Correct: The proposed split is directionally sound: the public API can remain key/`ComputeCtx` based while a lower VFS/index owns filesystem state. The current fs graph already centralizes disk access behind `FsBackend` (`crates/pixi_compute_fs/src/lib.rs:85`) and exposes the desired public keys (`FileMetadataKey`, `DirectoryListingKey`, `RecursiveDirectoryListingKey`, `GlobMTimeKey`) in one crate. The compute engine also already has daemon-facing invalidation/replacement hooks (`ComputeUpdater::changed_if_new`, `changed_to`, `commit_report` at `crates/pixi_compute_engine/src/engine.rs:242`, `:257`, `:279`).
- Correct: The performance target is realistic only if `glob_mtime(root, pattern)` stops materializing thousands of per-file compute nodes on the cold path. The current implementation recursively computes directory glob nodes and then computes `FileMetadataKey` for every matching file (`crates/pixi_compute_fs/src/lib.rs:306`, `:360`). A VFS aggregate can support the same public key while moving the high-cardinality work into an indexed data structure.
- Correct: The current code has equality/backdating hooks that can benefit from VFS `changed_to` updates. `Key::equality` is the cutoff mechanism (`crates/pixi_compute_engine/src/key.rs:127`), and dirty/rdep propagation is already separated between direct dirtying and value replacement (`crates/pixi_compute_engine/src/key_graph.rs:1024`, `:1035`, `:1048`).

- Fixed: None. This is plan review only.

- Blocker: The plan needs an explicit atomicity story between VFS commits and compute-graph invalidation. If the VFS mutates first and graph invalidation happens later, a compute request can observe new VFS data while the graph still serves old cached values, or vice versa. If graph invalidation happens first and VFS update lags, recomputation can cache old VFS data at the new graph version. Before implementation, define one commit protocol: either a single VFS owner applies an event batch and synchronously commits corresponding graph `changed`/`changed_to` updates before publishing the new VFS revision, or readers pin a VFS revision tied to the graph version they compute under.
- Blocker: Active glob `QueryState` lifecycle is under-specified. It is not enough to update currently-live query states unless the VFS knows they must stay registered for as long as the compute graph may return a cached `GlobMTimeKey`. If a query state is evicted while `GlobMTimeKey { root, pattern }` remains cached, later watcher events will not invalidate that cached key. The plan needs either no eviction initially, graph-to-VFS eviction notifications, or a conservative fallback that invalidates glob keys whose query state was dropped.
- Blocker: Snapshot/version behavior must be defined. Current graph versions are not true filesystem snapshots, but old graph snapshots can keep precomputed values. A mutable global VFS index can break this if a snapshot computes a key after the VFS has advanced: it may cache new VFS data under an old graph version. Decide whether this is acceptable and test it, or introduce VFS revision pinning for compute contexts.
- Blocker: Watcher overflow/rescan fallback has a race window unless events during rescan are handled. A full rescan that runs while new watcher events arrive can lose changes unless the design uses a changed-since token, pauses/queues events, marks the root dirty until replay is complete, or retries when the watcher reports an ambiguous state.
- Blocker: `read_file(path)` cannot simply read disk on demand while metadata comes from the VFS. That can return untracked content if the file changes between VFS stat and disk read. At minimum, `read_file` needs a content/version dependency from the VFS and a read-then-restat or stat-read-stat validation strategy; otherwise compute keys may cache bytes that no VFS event will precisely invalidate.

- Note: QueryState max ordering must preserve existing result semantics. Current `GlobMTime` tie-breaking chooses the lexicographically smaller `PathBuf` when mtimes are equal (`crates/pixi_compute_fs/src/lib.rs:395`). A `BTreeMap<(mtime, file_id), file_id>` is not equivalent unless `file_id` order is path order. Use `(mtime, reverse/path-order-key, file_id)` or explicitly compare current paths on ties.
- Note: Rename and path change handling is more complex than “changed file id”. Glob membership depends on both old and new paths. A file renamed out of a matching pattern must be removed from `matched_files` and `max_by_mtime`; a file renamed into a pattern must be added; a same-mtime rename may change the designated file because of path tie-breaking.
- Note: Directory keys are not automatically supported by active glob maintenance. `read_dir(path)` and `read_dir_recursive(path)` must be invalidated or replaced on membership/type changes. Current invalidation explicitly dirties file metadata/content plus the path and parent directory listings (`crates/pixi_compute_fs/src/lib.rs:493`, `:505`). A VFS event bridge must continue to emit affected `DirectoryListingKey` and ancestor `RecursiveDirectoryListingKey` updates, not just glob updates.
- Note: Metadata semantics need a representation for errors and absence. Current `metadata(path)` returns `exists=false` on not-found but propagates other IO errors (`crates/pixi_compute_fs/src/lib.rs:94`). `read_dir(path)` propagates read errors (`crates/pixi_compute_fs/src/lib.rs:113`). An index that only stores successful records will not reproduce permission errors, broken symlink behavior, transient IO failures, or missing directories unless error/unknown states are modeled.
- Note: Symlink traversal must be specified. Current listing classifies entries by `DirEntry::file_type()` and does not recursively traverse symlinked directories as directories (`crates/pixi_compute_fs/src/lib.rs:125`). A parallel walker must match that behavior unless the public semantics are intentionally changed.
- Note: Path identity needs a clear normalization policy. Existing keys use direct `PathBuf`s. A VFS path interner must not accidentally merge distinct keys (`./a` vs absolute path, case variants on case-insensitive filesystems, symlinked roots) or split equivalent watcher paths such that invalidation misses the compute key actually queried.
- Note: The proposed complexity `O(changed files × affected globs)` can still be too high with many active globs. A first implementation can accept this, but the plan should measure active glob count and memory. `matched_files` plus `max_by_mtime` duplicates per query can become large for broad patterns.
- Note: Cold scan should build a complete staged index and publish it atomically. Do not leave partially indexed roots visible to queries if the scan is canceled or errors. Also deduplicate concurrent `ensure_root_indexed(root)` calls so daemon request fanout does not launch multiple full scans.
- Note: Avoid holding a VFS write lock across disk IO or graph commits. Watcher updates, rescans, and compute reads will run concurrently. Holding locks across `.await` or graph calls risks deadlocks and latency spikes.
- Note: Invalidating only active globs means first query after a change is fine only if the VFS index itself is current. If a root is dirty/unknown, `glob_mtime` should either wait for repair/rescan or return a domain error; it should not silently answer from stale index data.
- Note: `changed_to` is a useful bridge, but only for keys whose new values the VFS can compute exactly. For broad changes, it may be safer to call `changed` and let the key recompute from a repaired VFS revision. Mixing both paths needs a clear rule so same event batches do not double-invalidate or miss backdating opportunities.
- Note: Public introspection will change. Today glob internals expose directory and metadata dependencies (`DirectoryGlobMTimeKey` and `FileMetadataKey`). A VFS aggregate will hide per-file deps from compute introspection unless supplemental introspection is added. That may be acceptable, but it is a semantic/API tradeoff to document.

## Minimal experiments before committing

1. Build a read-only VFS prototype populated by a parallel scan, with no watcher support. Measure: scan time, index memory, first `glob_mtime` over the index, warm `glob_mtime`, and compare results against the current implementation and `pixi_glob` on the same fixture.
2. Add an in-memory active `GlobQueryState` prototype and feed synthetic update events. Measure update time for: nonmatching file mtime change, matching file mtime change, create latest match, delete latest match, rename into/out of pattern, and same-mtime path tie changes.
3. Measure many active globs: e.g. 1, 10, 100, 1000 broad patterns over the same root. Record update latency and memory. This validates or falsifies the `changed files × affected globs` assumption.
4. Prototype the VFS-to-graph bridge for only `GlobMTimeKey` using `changed_to`. Test that a dependent key does not recompute when the aggregate value is equal, and does recompute when the designated file or mtime changes.
5. Add a race test with concurrent `glob_mtime` requests while a VFS event batch commits. Assert readers see either the old graph/VFS pair or the new graph/VFS pair, not a mixed state.
6. Add an overflow/rescan simulation: mark root dirty, mutate files during rescan, replay queued events, and compare the final index/glob results with a fresh full scan.
7. Add `read_file` validation experiments: modify a file during read, modify content with unchanged mtime/size if possible, and verify the cache is invalidated or the read retries.
8. Add semantic comparison fixtures for symlinks, permission errors, missing directories, file-to-directory and directory-to-file transitions, case-only renames where supported, and relative vs absolute roots.

## Recommendation

Do not commit to the full VFS architecture until experiments 1-5 show correctness and stable performance. The architecture is promising, but the risky part is not the in-memory index itself; it is maintaining exact graph invalidation semantics while the index mutates concurrently with daemon reads. Start with a narrow `GlobMTimeKey` VFS prototype and an explicit VFS revision/graph commit protocol, then expand to `metadata`, `read_dir`, `read_dir_recursive`, and `read_file` only after those semantics are proven.
